## Design Context

### Users

Solo developers managing their own issues and agent tasks across projects. They
open Beads UI to quickly triage what's ready, check what agents are working on,
and move issues through the pipeline. Context is fast, focused work sessions —
not long exploratory browsing. They're already comfortable with the terminal and
`bd` CLI; the UI is a visual complement, not a replacement.

### Brand Personality

**Fast, sharp, technical.** This is a power-user tool that respects the
developer's intelligence and time. It should feel like a precision instrument —
responsive, dense, and deliberate. Every pixel earns its place.

### Emotional Goals

- **Confidence & control** — The user should always know exactly what's
  happening across their projects. No ambiguity, no hidden state. Information
  hierarchy is clear at a glance.
- **Speed & efficiency** — Triage 20 issues in 2 minutes. The UI never makes you
  wait, never makes you hunt. Keyboard-first, minimal clicks.
- **Delight & craft** — Small moments of quality (smooth transitions, tight
  alignment, considered typography) that signal this tool was made with care.
  Not flashy — quietly excellent.

### Aesthetic Direction

**Reference**: Linear — keyboard-first, fast, dense but polished. The gold
standard for developer-facing issue trackers. Beads UI should feel like it
belongs in that tier: clean, fast, information-dense without feeling cluttered.

**Anti-references**:

- NOT barebones/ugly-functional — it should never look unfinished or like a side
  project. Craft matters.
- NOT colorful/playful — no confetti, no illustrations, no bubbly rounded
  components. No Monday.com energy.

**Theme**: Light and dark mode (both supported). Dark theme uses deep navy
(`#0b1021`) — not pure black. System font stack, no custom fonts. Color is used
sparingly and semantically: status indicators, type badges, links. The palette
is neutral-first with purposeful color accents.

**Existing design system** (from codebase):

- 2px baseline spacing scale (`--space-1` through `--space-10`)
- `color-mix()` for derived badge/state colors
- Frosted glass sticky header (`backdrop-filter: blur`)
- All custom components (no library) — tabs, badges, selects, toasts, dialogs,
  board columns
- No brand assets yet (no favicon, no logo, no SVGs)

### Design Principles

1. **Density over decoration** — Show more information in less space. Prefer
   compact rows, tight spacing, and data-rich views over whitespace-heavy
   layouts. Every element should carry information.

2. **Speed is a feature** — Perceived and actual performance matter equally.
   Instant feedback on interactions, no unnecessary loading states, keyboard
   shortcuts for everything. If it can be done without a modal, skip the modal.

3. **Quiet confidence** — The interface shouldn't shout. Use restraint with
   color, animation, and visual weight. Let the content (issues, statuses,
   priorities) be the loudest thing on screen. Chrome should recede.

4. **Craft in the details** — Alignment, consistent spacing, smooth transitions,
   proper focus states. The difference between "good enough" and "excellent"
   lives in the small things. Ship polish, not features.

5. **Semantic color, not decorative** — Color always means something: status,
   priority, type, or state change. Never use color for pure decoration. This
   keeps the interface scannable and accessible.
